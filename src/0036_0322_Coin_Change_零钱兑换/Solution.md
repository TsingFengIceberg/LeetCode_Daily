# 算法复盘笔记：LeetCode 322. 零钱兑换 (Coin Change)


---

## 🚫 踩坑实录与思维转折 (极其珍贵的试错经验)

### ❌ 我的最初弯路：贪心直觉法 (Greedy)
**我的原思路（一字不落记录）**：“我没看提示，是排序，然后先组大然后插空弄小的？”

**面试官一针见血的破局反例**：
在现实生活中找零确实是“先挑大面额，再用小面额填缝”，但在算法题中，硬币面值如果没有严格的倍数关系，贪心会引发毁灭性打击：
* **反例 1（错失最优解）**：`coins = [1, 3, 4]`, `amount = 6`。
  按我的思路：拿最大的 `4`，剩 `2` 用两个 `1` 填补。总计 **3 枚** (4+1+1)。
  真实最优解：直接拿两个 `3`！总计 **2 枚** (3+3)。
* **反例 2（误报无解）**：`coins = [2, 3]`, `amount = 7`。
  按我的思路：拿两个 `3`，剩 `1`，发现没有 `1` 元硬币，**得出结论：凑不出，返回 -1**。
  真实最优解：拿一个 `3` 和两个 `2`！总计 **3 枚** (3+2+2)。

**结论**：局部最优（选最大硬币）不仅无法推导出全局最优，甚至会把系统逼上死路。必须放弃走一步看一步的贪心，拥抱统筹全局的**动态规划 (DP)**。

---

## 💡 正确解法引导：动态规划 (Dynamic Programming)
既然不能贪心，我们就采用“从底向上查字典”的填表法。
1. **明确状态 (`dp` 数组定义)**：`dp[i]` 表示凑齐金额 `i` 所需要的**最少**硬币个数。
2. **防坑初始化 (极大值)**：我们要求“最小值”，所以除了 `dp[0] = 0`，其他金额默认填上一个“不可达的极大值”。
   * **死亡陷阱**：绝不能用 `Integer.MAX_VALUE`，否则后续执行 `dp[i-coin] + 1` 时会发生整型溢出变成负数！
   * **绝佳替代品**：`amount + 1`。因为哪怕全用 1 元硬币，最多也就需要 `amount` 枚。`amount + 1` 在本题中等价于安全的无穷大 $\infty$。
3. **状态转移方程 (核心魔法)**：
   对于金额 `i`，遍历手里所有的硬币 `coin`。只要当前硬币面额没超过 `i`（`i - coin >= 0`）：
   `dp[i] = Math.min(dp[i], dp[i - coin] + 1)`
   （翻译：要么保持原方案，要么退回到金额 `i-coin` 的最优方案，再加上手里的这 1 枚硬币，两者取最小。）
4. **结局验收**：最后检查 `dp[amount]`。如果它还是初始的 `amount + 1`，说明大结局依然无解，按题意返回 `-1`；否则返回真实数值。

## 💻 工业级代码 (纯净版 DP 模板)

```java
import java.util.Arrays;

class Solution {
    public int coinChange(int[] coins, int amount) {
        // 创建 dp 数组，长度为 amount + 1 (因为包含了金额 0)
        int[] dp = new int[amount + 1];
        
        // 初始化为一个安全的极大值 (相当于正无穷)
        Arrays.fill(dp, amount + 1);
        
        // Base Case：凑金额 0 需要 0 枚硬币
        dp[0] = 0;
        
        // 外层循环：从金额 1 一直推导到金额 amount
        for (int i = 1; i <= amount; i++) {
            // 内层循环：尝试手里的每一枚硬币
            for (int coin : coins) {
                // 如果当前金额 i 装得下这枚硬币
                if (i - coin >= 0) {
                    // 状态转移：打擂台求最少硬币数
                    dp[i] = Math.min(dp[i], dp[i - coin] + 1);
                }
            }
        }
        
        // 大结局验收：如果没被更新过，说明凑不齐，返回 -1
        return dp[amount] > amount ? -1 : dp[amount];
    }
}

```

### 📊 复杂度分析

* **时间复杂度**: $O(\text{amount} \times N)$。$N$ 为硬币面值的种类数。我们需要计算 `amount` 个状态，每个状态需要遍历 $N$ 个硬币进行转移。
* **空间复杂度**: $O(\text{amount})$。维护了一个长度为 `amount + 1` 的一维 `dp` 数组。

## 🏗️ 真实大厂工程实例深度剖析

把题目中的“硬币”换成“资源”，这就是大厂底层资源调度最核心的**完全背包问题**：

1. **云原生调度 (K8s/阿里云)**：当用户申请 `51` 核的分布式计算资源时，底层有 `[4, 8, 16, 24, 32]` 等固定核数的虚拟机规格。底层调度引擎运行此 DP 算法，既不浪费算力，又能用**最少数量的机器节点**凑齐资源。机器节点越少，大数据计算时的跨机器网络通信（Shuffle）延迟就越低！
2. **电商营销中台最优发券**：大促时给高净值用户发放总额 `60` 元的回馈金，可用面额库为 `[20, 30, 45]`。发券接口调用 DP，精准算出发 2 张 `30` 元券的最优解。相比于贪心（发更多零碎券），极大地减少了核心数据库 Coupon 表的高并发 `INSERT` 写入压力，防止大促期间连接池打爆。
