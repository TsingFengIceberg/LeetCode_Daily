# [0238. 除自身以外数组的乘积 (Product of Array Except Self)]

## 💡 核心思想：左右分治与“前缀/后缀积”
这道题的核心限制在于**“不能使用除法”**，直接封死了“求总乘积再除以自身”的偷懒路线。
破局点在于**切割视线**：对于任何一个元素 `nums[i]`，除了它自身的乘积，实际上等于 **(严格在它左侧所有元素的乘积) * (严格在它右侧所有元素的乘积)**。借此，我们可以利用前缀和（积）的思想，通过正反两次遍历来解决问题。

## 🚨 踩坑实录与代码坏味道（一字不差的答疑与纠错记录）
在尝试使用 `O(N)` 空间复杂度的基础解法时，你写出了一版存在严重逻辑漏洞和工程坏味道的代码，这也是大厂笔试中最容易被刷掉的典型反面教材：

* **致命错误 1：Java 数组默认值陷阱**
    * 你的原代码：`for (int i = 1; i < prefix.length; i++){ prefix[i] *= prefix[i - 1]; }`
    * **纠偏**：在 Java 中，`new int[]` 刚初始化时，所有元素默认值全是 `0`。你用 `0 *= 0` 去滚动累乘，导致整个 `prefix` 数组从头到尾全变成了 `0`。正确做法是必须引入原数组 `nums` 的值参与计算：`prefix[i] = nums[i] * prefix[i - 1];`。
* **致命错误 2：典型的“复制粘贴后遗症” (Copy-Paste Error)**
    * 你的原代码（在计算 suffix 后缀积的循环里）：`for (int i = suffix.length - 2; i >= 0; i--){ prefix[i] *= prefix[i + 1]; }`
    * **纠偏**：明明是在反向遍历计算 `suffix` 数组，但循环体里被修改的变量居然还是 `prefix`！这种低级失误在真实面试中极其扣分，说明写代码时处于无意识的机械复制状态。
* **坏味道 3：边缘处理极其痛苦混乱**
    * 你的原代码试图用硬编码特判：`prefix[0] = suffix[0]; suffix[suffix.length - 1] = prefix[prefix.length - 1];`
    * **纠偏**：这是因为你对状态的定义不够优雅。如果你定义 `prefix[i]` 是“包含 `nums[i]` 自身的乘积”，首尾处理就会很别扭。大厂常用的技巧是**“错位定义法”**：让 `prefix[i]` 表示**严格在 `nums[i]` 左侧**的连乘积。由于 `nums[0]` 左边没元素，直接初始化为乘法单位元 `1`（或者在后续 O(1) 解法中巧妙避开越界），这样结算公式直接变成 `result[i] = prefix[i] * suffix[i]`，彻底消灭首尾特判的补丁代码。

## 🚀 进阶突破：O(1) 空间复杂度的极限压榨
在解决了上述 Bug 后，你成功完成了进阶挑战，写出了极其惊艳的 $O(1)$ 空间解法（不包含返回数组）：
1.  **复用数组**：直接拿要求返回的 `result` 数组来充当 `prefix` 数组，省去了一个 $O(N)$ 数组的开销。
2.  **变量流式滚动**：由于后缀积是反向从后往前算的，我们根本不需要一个完整的 `suffix` 数组来全量存储。直接用一个 `int temp = 1;` 变量，在反向遍历时，**边维护后缀积，边乘以 `result` 里存好的前缀积**，直接原地输出最终答案！

## 🏭 工业界真实工程实例：AI 推荐系统中的“目标泄露”与矩阵除法灾难
这套算法骨架绝不仅仅是做题，它在字节跳动/阿里的机器学习平台中有极其硬核的落地场景。
在构建用户画像时，算法工程师经常要做 **Leave-One-Out (留一法)** 特征聚合。比如用户浏览了 A, B, C, D，在预测 C 时，必须聚合 A, B, D 的行为特征。
* **为什么不用除法（总特征除以当前特征）？** 因为真实的特征往往是高维 Embedding 向量或复杂的交叉特征矩阵。**矩阵是没有通用除法的（求逆复杂度极高且不一定可逆）**。一旦发生“目标泄露 (Target Leakage)”，即预测 C 时把 C 自己的行为也算了进去，模型在线下表现会极好，一上线就会引发灾难级事故。
* **架构落地**：在 Flink 实时流处理中，为了节约极其宝贵的内存资源（一台机器要处理海量用户），底层 UDAF 算子就会采用你写的这套解法：正向算出前缀特征矩阵并缓存，反向流式用一个 State (对应你的 `temp` 变量) 边滚动边正交融合，最终在不使用除法、不增加内存开销的前提下，完美输出剔除当前元素的特征集。

## 最终 Java 代码实现
```java
class Solution {
    public int[] productExceptSelf(int[] nums) {
        // 1. 直接复用最终的返回数组 result 作为前缀积的存储容器，满足 O(1) 额外空间要求
        int[] result = new int[nums.length];
        
        // 2. 初始化前缀积的基座：此时 result[0] 代表包含 nums[0] 在内的连乘积
        result[0] = nums[0];     
        
        // 3. 正向遍历，把 result 数组填满前缀积
        for (int i = 1; i < result.length; i++){
            result[i] = nums[i] * result[i - 1];
        }
        
        // 4. 极致的内存压榨核心：用一个动态变量 temp 替代整个 suffix 后缀积数组
        // temp 代表了“严格在当前指针 i 右侧的所有元素的乘积”
        int temp = 1;
        
        // 5. 反向遍历，边算后缀积边结算最终答案
        for (int i = result.length - 1; i > 0; i--){     
            // result[i - 1] 恰好是严格在 i 左侧的前缀积
            // temp 恰好是严格在 i 右侧的后缀积
            result[i] = result[i - 1] * temp;
            
            // 结算完当前位置后，立刻更新 temp，把当前的 nums[i] 卷进去，供下一个位置使用
            temp *= nums[i];
        }
        
        // 6. 边界打补丁：处理索引 0 的情况
        // i = 0 时左侧没有元素，所以它的答案直接等于右侧所有元素的连乘积，也就是遍历结束后的 temp
        result[0] = temp;
        
        return result;
    }
}
