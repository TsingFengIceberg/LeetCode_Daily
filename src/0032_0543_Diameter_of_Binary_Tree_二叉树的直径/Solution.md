# 算法复盘笔记：LeetCode 543. 二叉树的直径 (Diameter of Binary Tree)

## 💡 核心思路：深度与直径的降维转换
这道题最大的陷阱在于：“最长路径可能根本不经过根节点”。


**破局关键（大厂标准套路）：后序遍历 + 全局记分牌**
1. **职责分离**：递归函数 `getDepth(node)` 的唯一职责是——计算并向上级返回“以当前节点为根的子树的最大深度（边数）”。公式为 `max(左深度, 右深度) + 1`。
2. **暗中操作**：在每次计算出左右子树深度即将返回之前，我们利用这两个深度，顺手计算一下**“穿过当前节点的最长路径边数”**，公式为 `左深度 + 右深度`。
3. **全局打擂台**：将算出的“当前路径边数”与全局变量 `maxDiameter` 进行 PK，永远保留最大的那个。一遍遍历结束，全局最大直径自然浮出水面。

---

## 💻 工业级代码与详尽注释

```java
public class Solution {
    // 【工程选型】定义全局变量作为“记分牌”，突破 Java 单一返回值的限制
    private int maxDiameter = 0;

    public int diameterOfBinaryTree(TreeNode root) {
        // 触发自底向上的后序遍历
        getDepth(root);
        // 遍历结束后，记分牌上保留的就是整棵树的全局最大直径
        return maxDiameter;
    }

    // 辅助函数：计算并返回以 node 为根的子树的最大深度（节点数/边数逻辑统一）
    private int getDepth(TreeNode node) {
        // Base Case：空节点的深度为 0
        if (node == null) {
            return 0;
        }

        // Divide：分别探底，向左右小弟索要它们的最大深度
        int leftDepth = getDepth(node.left);
        int rightDepth = getDepth(node.right);

        // 【核心魔法：顺手牵羊更新直径】
        // 穿过当前节点的最长路径边数，恰好等于“左子树深度 + 右子树深度”
        // 我们在这里拦截一下数据，更新全局最大值
        maxDiameter = Math.max(maxDiameter, leftDepth + rightDepth);

        // Conquer：向上级领导汇报当前节点自己的最大深度
        return Math.max(leftDepth, rightDepth) + 1;
    }
}

```

### 📊 复杂度分析

* **时间复杂度**: $O(N)$。完美避开了重复计算，每个节点在后序遍历中被精准访问且仅访问一次。
* **空间复杂度**: $O(H)$。$H$ 为二叉树的高度。主要开销为递归造成的系统调用栈深度。最坏情况（退化成链表）为 $O(N)$，最好情况（平衡树）为 $O(\log N)$。

## 🏗️ 真实大厂工程实例深度剖析

这绝不是一道简单的数学题，在大型分布式架构中，“树的直径”代表着系统的“最远通信延迟 / 最坏情况”。

### 1. 字节跳动/腾讯视频：P2P 与 CDN 边缘节点网络拓扑

* **业务痛点**：视频数据的 P2P 分发网络是一棵庞大的生成树。如果树的“直径”过大，最远两端的边缘路由器通信连跳数（Hop）过多，直接导致视频首帧卡顿。
* **架构落地**：中控系统定期运行 $O(N)$ 的直径计算算法扫描全网。一旦 `maxDiameter` 超过业务阈值，强行触发拓扑重构（压扁树结构），保障亿级用户的毫秒级播放体验。

### 2. 阿里云/分布式数据库：Gossip 协议的心跳收敛时间

* **业务痛点**：在无中心化的 Redis Cluster 等集群中，节点宕机消息靠 Gossip（八卦）协议在树状拓扑中口口相传。
* **架构落地**：架构师必须精准计算出集群通信树的“最大直径”，以此推算消息传遍全网的最长耗时。这个数值直接决定了中间件 `cluster-node-timeout`（脑裂超时重选举）的生死配置参数。
