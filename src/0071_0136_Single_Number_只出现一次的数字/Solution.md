# [0136. 只出现一次的数字 (Single Number)]

## 💡 思路演进：从常规集合到 $O(1)$ 空间的极限施压
在最初拿到题目时，我们首先想到的是使用 Java 的 `HashSet`。利用哈希表去重的特性，遍历数组，存在则 `remove()`，不存在则 `add()`，最后留在集合里的就是唯一的数字。
**痛点分析**：这种做法虽然时间复杂度是 $O(N)$，但空间复杂度也是 $O(N)$，在面临千万级数据时会导致大量内存开销，直接违背了题目要求的“常量额外空间”限制。在真实面试中，如果止步于此，通常只能拿到及格分。

## 🧠 核心破局点：异或 (XOR) 运算的降维打击
为了将空间复杂度严格压缩到 $O(1)$，我们引入了底层的位运算——**异或（XOR，`^`）**。
异或运算（不进位加法）完美契合本题的三个数学定律：
1. **归零律**：任何数和其自身异或，结果为 0（即 $a \oplus a = 0$）。
2. **恒等律**：任何数和 0 异或，保留原值（即 $a \oplus 0 = a$）。
3. **交换律与结合律**：运算顺序打乱完全不影响结果。

利用这些特性，我们将数组中所有元素连起来做全局异或。成对出现的数字在底层二进制层面会互相“抵消”变成 0，而那个落单的数字与 0 异或，最终原形毕露。这种做法不需要借助任何额外的数据结构，仅用一个 `int` 变量即可完成状态收集。

## 🏭 工业界真实工程实例：架构师视角的异或
这绝不是一道简单的“玩具题”，异或在海量数据和高可用架构中有着不可替代的工业级应用：
1. **RAID 5 磁盘阵列容灾**：底层分布式存储中，利用异或生成“校验盘”。当任意一块装载数据的硬盘物理损坏时，系统不需要 1:1 的昂贵备份，只需将存活的硬盘与校验盘再次进行异或操作，即可瞬间“算”出丢失的千万级数据。
2. **微服务海量流水对账**：在千万级订单/支付微服务异步同步时，如果因为网络抖动漏掉一笔订单，无需将上 GB 的数据加载进内存比对（极易引发 OOM）。双方服务只需在内存中维护一个极小的异或 Checksum，对账时将两边的 Checksum 互相异或，非 0 结果就是那笔丢失的订单 ID，真正做到了 $O(1)$ 空间复杂度的架构级降本增效。

## 🎯 大厂面试 Follow-up 变形预测
在字节、阿里等一线大厂面试中，手撕完这道题后，面试官通常会紧接着抛出以下进阶变种：
- **变种 1：LeetCode 137. 只出现一次的数字 II**（除了某个元素只出现一次以外，其余每个元素均出现**三次**）。
  *思路拓展*：异或无法处理出现三次的情况。需要改用统计所有数字在每个二进制位上 1 的个数，然后对 3 取模。
- **变种 2：LeetCode 260. 只出现一次的数字 III**（有**两个**元素只出现了一次，其余出现两次）。
  *思路拓展*：全局异或后得到的是两个只出现一次的数字的异或和。需要利用 `x & (-x)` 提取出最低位的 1，将数组强行分为两组，把问题降维转化为两道 136 题。

## 最终 Java 代码实现
```java
class Solution {
    /**
     * 寻找数组中唯一只出现一次的数字
     * 时间复杂度: O(N) - 仅需遍历数组一次
     * 空间复杂度: O(1) - 仅使用一个 int 型变量
     */
    public int singleNumber(int[] nums) {
        // 初始化为 0，因为 0 ^ a = a，可以完美接收第一个数
        int result = 0; 
        for (int num : nums) {
            // 利用成对数字异或为 0 的特性，逐个抵消
            result ^= num; 
        }
        // 最终留下的就是那个没有被抵消的落单数字
        return result; 
    }
}
