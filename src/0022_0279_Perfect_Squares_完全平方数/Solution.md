# 算法复盘笔记：LeetCode 279. 完全平方数 (Perfect Squares)

## 一、 题目剖析与大厂考点
* 考察频次：⭐⭐⭐⭐⭐ (动态规划/完全背包问题的开山鼻祖，字节、腾讯高频题)
* 核心标签：动态规划 (Dynamic Programming)、完全背包、广度优先搜索 (BFS)、数学
* 时间/空间复杂度要求：时间 $O(n \sqrt{n})$，空间 $O(n)$。
* 题目痛点：面对找零钱/凑数字问题，容易陷入“贪心算法”的陷阱（比如每次都尽量取最大的平方数，但在某些特殊数据下贪心会得不到最优解），必须使用动态规划来遍历所有状态空间。

## 二、 核心原理解析：动态规划与“打擂台”机制 (极其详细版)

这道题是理解动态规划中“状态转移”与“最优化”思想的最佳教材。

### 1. 明确状态定义与初始化边界
* **状态定义**：定义一个一维数组 `dp[]`，`dp[i]` 的物理含义是：**凑成数字 i 所需要的最少完全平方数的数量**。
* **初始化边界**：
  * `dp[0] = 0`。因为凑成数字 0 不需要任何完全平方数，这是所有状态推导的基石。
  * `dp[1]` 到 `dp[n]` 初始化为 `Integer.MAX_VALUE`（或 `i` 本身）。因为后续我们需要寻找“最小值”，如果初始化为 0，就会干扰 `Math.min` 的判断，导致结果永远是 0。

### 2. 状态转移方程的深度推导（为什么必须套 Math.min？）
以求 `dp[12]` 为例，我们能用的平方数只有 $1^2=1$, $2^2=4$, $3^2=9$。
要想凑出 12，我们倒推最后一步，只有三种合法的来源：
* 从 11 走过来：消耗一个 1。总步数为 `dp[12 - 1] + 1 = dp[11] + 1`
* 从 8 走过来：消耗一个 4。总步数为 `dp[12 - 4] + 1 = dp[8] + 1`
* 从 3 走过来：消耗一个 9。总步数为 `dp[12 - 9] + 1 = dp[3] + 1`

**为什么必须套用 `Math.min`？**
因为代码是通过 `for` 循环挨个尝试上述三种情况的。如果不加 `Math.min`：
1. 第一回合：尝试用 1，算出 `dp[12] = dp[11] + 1` (假设等于 4)，写入 `dp[12]`。
2. 第二回合：尝试用 4，算出 `dp[12] = dp[8] + 1` (假设等于 3)。如果不作比较直接赋值，刚才的 4 就被**直接覆盖**了。
3. 第三回合：尝试用 9，算出 `dp[12] = dp[3] + 1` (假设等于 4)。如果不作比较直接赋值，最优解 3 就被抹杀，最终得到了错误结果 4。

因此，**核心状态转移方程**必须是：
`dp[i] = Math.min(dp[i], dp[i - j * j] + 1);`
这句话的业务翻译是：“凑成数字 i 的最少个数，等于 **【目前已知的历史最优解】** 和 **【刚刚尝试的这种新方案的个数】** 这两者之间的较小值。”通过不断地打擂台，最终活下来的才是真正的最优解。

## 三、 核心代码实现 (Java)

```java
class Solution {
    public int numSquares(int n) {
        int[] dp = new int[n + 1];
        dp[0] = 0; // 地基
        
        // 预填最大值，防止打擂台时被初始化的 0 干扰
        for (int i = 1; i <= n; i++) {
            dp[i] = Integer.MAX_VALUE;
        }
        
        // 遍历所有需要求解的数字 i
        for (int i = 1; i <= n; i++) {
            // 尝试所有可能的完全平方数 j*j
            for (int j = 1; j * j <= i; j++) {
                // 状态转移：历史最优解 vs (退回到上一步的最优解 + 1步)
                dp[i] = Math.min(dp[i], dp[i - j * j] + 1);
            }
        }
        
        return dp[n];
    }
}

```

## 四、 工业界真实场景映射 (工程实例深度剖析)

这不仅是一道算法题，本质上它是一个**“资源最优化分配引擎”**。在工业界，这种分配逻辑被广泛应用：

### 1. 视频编解码引擎（H.265 / HEVC）的宏块划分优化

* **业务痛点**：在视频流传输中，画面会被切分成正方形的像素块。每个块都需要在文件头记录坐标，块数量越多，视频文件就越大。为了节省带宽，编码器必须用**最少的块**填满一个复杂区域。
* **算法映射**：编码器底层就是在这套 DP 算法。给定一个不规则的高维残差区域面积 $N$，系统通过动态规划计算出如何用最少数量的完全平方数（如 $4\times4, 8\times8, 16\times16$ 的正方形块）去完美覆盖它。通过局部最优推导全局最优，极大地提高了视频压缩率。

### 2. 分布式存储系统与 GPU 显存分配的“非线性内存池”

* **业务痛点**：在高性能数据库（如 Redis）或深度学习矩阵计算中，如果频繁申请不规则的内存，会造成严重的内存碎片。因此，系统通常预先申请并维护多个**完全平方数大小（1MB, 4MB, 9MB, 16MB）**的定长内存池。
* **算法映射**：当一个高并发请求申请 12MB 内存时，内存调度器如果直接给 16MB 会产生内部碎片；如果给 12 个 1MB，会产生过多的指针维护开销。调度器瞬间执行类似 DP 的分配逻辑，得出 $12 = 4 + 4 + 4$。从而极其精准地分配 3 个 4MB 的块，既做到了零碎片，又将指针数量（资源开销）压到了最低。

### 3. 数学界的降维打击 (拉格朗日四平方和定理)

这道题有一个空间复杂度 $O(1)$ 的数学极客解法：拉格朗日四平方和定理证明了**任何一个正整数最多只需要 4 个完全平方数就能凑出**。结合勒让德三平方和定理判断，可以直接把答案锁定在 1, 2, 3, 4 之中。但这属于纯数论范畴，在通用业务开发中，动态规划依然是解决此类背包问题的最核心利器。
