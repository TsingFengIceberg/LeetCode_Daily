# 算法复盘笔记：LeetCode 45. 跳跃游戏 II (Jump Game II)


---

## 🚫 踩坑实录与思维转折 (极其珍贵的试错经验)

### ❌ 我的最初弯路：逆向贪心法 (Backward Greedy)
**我的原思路**：先遍历一遍记录下 `maxjump` 数组（每个下标最远能跳到哪里）。然后从 `maxjump[n-1]` 开始往回遍历，第一次遇到“够不到”当前目标位置时，就把刚刚那个“能挂到”的位置设为新目标，跳跃次数加一，依此类推直到开头。

**面试官一针见血的破局反例**：
如果测试用例是 `nums = [4, 0, 0, 1, 4]`，终点是 `target = 4`。
按照逆向找“第一个够不到”的逻辑：
1. `i = 3` 时，能到 4。
2. `i = 2` 时，最远只能到 2，**第一次遇到够不到！**
3. 于是把 `i = 3` 作为新目标（跳跃 1 次）。再往前找，发现 `i = 0` 能到 3（跳跃 1 次）。总计 **2 次**。
**致命漏洞**：在 `i = 0` 时，最远能跳到 4，明明可以**一步登天**！
逆向寻找“第一个断层”的逻辑，建立在 `maxjump` 数组是严格单调递增的错误假设上。如果有前面能力极强、能直接跨越断层的“超级跳板”，这种逻辑就会完美错过最优解。

---

## 💡 正确解法引导：正向贪心 (Forward Greedy)
放弃复杂的逆向和 $O(N^2)$ 的回溯，我们采用“势力范围”与“暗中侦察”的正向思维：
1. **势力范围 (`currentEnd`)**：记录当前这一跳，最远能覆盖到哪里。
2. **暗中侦察 (`currentMaxPosition`)**：在当前势力范围内一步步往前走时，利用 `i + nums[i]` 不断刷新我们下一步能到达的最远距离。
3. **被迫起跳**：当我们走到当前势力范围的边界 (`i == currentEnd`) 时，说明这一跳的红利吃干抹净了，必须起跳 (`jumps++`)，并把势力范围更新为刚刚侦察到的最远距离。



---

## 💻 工业级代码与详尽注释 (1ms 极速版)

```java
class Solution {
    public int jump(int[] nums) {
        // 记录最终需要返回的最小跳跃次数
        int jumps = 0;
        
        // 记录“当前这一跳”所能覆盖的势力范围的右边界
        int currentEnd = 0;
        
        // 记录在当前势力范围内走动时，暗中侦察到的“下一步能跳到的最远位置”
        int currentMaxPosition = 0;
        
        // 【极其关键的边界防范】：遍历到倒数第二个元素即可 (nums.length - 1)
        // 题目保证一定能到达终点。如果不减 1，当 i 刚好等于终点且刚好走到边界时，会多算一次错误起跳。
        for (int i = 0; i < nums.length - 1; i++) {
            
            // 核心动作 1：暗中侦察
            // 在当前的势力范围内往前走，不断刷新下一步能跳到的最远位置纪录
            currentMaxPosition = Math.max(currentMaxPosition, i + nums[i]);
            
            // 核心动作 2：被迫起跳
            // 碰到当前这一跳的边界 currentEnd 时，说明必须交出下一次跳跃次数了
            if (i == currentEnd) {
                jumps++; // 跳跃次数 +1
                currentEnd = currentMaxPosition; // 势力范围扩张到新侦察到的最远距离
            }
        }
        
        return jumps;
    }
}

```

### 📊 复杂度分析

* **时间复杂度**: $O(N)$。仅需正向遍历数组一次，性能压榨到极致。
* **空间复杂度**: $O(1)$。只维护了三个整型状态变量，无额外内存开销。

## 🏗️ 真实大厂工程实例深度剖析

这不是一道纯粹的脑筋急转弯，而是底层网络和分布式系统用来“省资源”的核心算法：

1. **IoT 物联网/无人机组网的“最小路由跳数”**：在电量极度受限的无线传感器网络中，每次数据中转（广播）都会大幅消耗电池。底层通信协议利用此贪心算法，寻找跳数最少的路由路径，最大化延长整个硬件网络的生命周期。
2. **大型分布式微服务的跨版本平滑升级**：当核心中间件需要从 `v1` 升级到 `v100`，且存在各种不同跨度的兼容性补丁时。CI/CD 发布系统利用此算法计算出“停机次数（跳跃次数）”最少的升级跳板路径，极大减少业务中断时间。
