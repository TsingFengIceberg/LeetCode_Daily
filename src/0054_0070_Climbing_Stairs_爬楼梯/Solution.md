# [0070. 爬楼梯 (Climbing Stairs)]

## 💡 算法灵魂：动态规划 (DP) 的逆向思维
面对“每次爬 1 阶或 2 阶，求到达第 $N$ 阶的总方法数”这种极具迷惑性的题目，如果正向推导（DFS 画决策树），会陷入极度恐怖的指数级重复计算中。
**动态规划的破局点在于“逆向拆解”**：
想象你站在终点（第 $N$ 阶）的前一秒，你只能是从第 $N-1$ 阶跨了一步上来，或者是从第 $N-2$ 阶跨了两步上来。
因此，到达第 $N$ 阶的方法总数，必定等于到达 $N-1$ 阶与到达 $N-2$ 阶的方法数之和。
这就得出了 DP 的绝对核心——**状态转移方程**：$F(N) = F(N-1) + F(N-2)$。（这本质上就是一个斐波那契数列）。

## 🚀 大厂微操与架构级优化：状态压缩 (滚动数组)
1. **$O(N)$ 空间的直觉解法**：最朴素的 DP 写法是开辟一个 `int[] dp = new int[n + 1]` 数组，把从 1 到 $n$ 的所有历史状态全存下来。
2. **$O(1)$ 空间的极致降维**：在真实的工业界，内存是极其宝贵的资源。既然 $F(N)$ 永远只依赖于它前面的**两个**状态，那么更早的历史数据就成了毫无意义的内存垃圾。我们只需使用三个变量（`prev1`, `prev2`, `current`）交替滚动更新，即可将原本 $O(N)$ 的空间复杂度瞬间拍扁到绝对的 $O(1)$。这就是大厂极客必会的“状态压缩 / 滚动数组”技巧。

## 🏢 工业界真实工程实例：分布式防雪崩与底层视频解码
不要小看这个类似“做加法”的斐波那契逻辑，它在亿级系统中无处不在：
1. **微服务 RPC 框架的 Fibonacci Backoff (防雪崩退避策略)**：当核心服务宕机，为了防止全网重试流量引发“惊群效应”导致二次雪崩，中间件（如 gRPC）会采用斐波那契序列来决定每次重试的等待时间间隔（1秒, 2秒, 3秒, 5秒...）。这比指数退避更平滑，极大地保护了分布式系统的韧性。
2. **视频编解码 (H.264/H.265) 的底层状态预测**：在高清视频传输时，解码器不会逐张下载全量图片，而是利用动态规划思想，通过参考前一帧（P 帧）或前两帧（I 帧）的基础状态，配合微小的数据增量来还原当前画面，从而节省下惊人的网络带宽。

## 💻 最终 Java 代码实现 (极致空间压缩版)

```java
class Solution {
    public int climbStairs(int n) {
        // 边界条件：1 阶有 1 种方法，2 阶有 2 种方法
        if (n <= 2) {
            return n;
        }
        
        // 极致降维：抛弃 O(N) 的 dp 数组，仅使用三个变量交替滚动
        int prev2 = 1; // 相当于 dp[i-2]
        int prev1 = 2; // 相当于 dp[i-1]
        int current = 0; // 相当于 dp[i]
        
        // 自底向上，从第 3 阶开始推导
        for (int i = 3; i <= n; i++) {
            current = prev1 + prev2; // 状态转移方程
            
            // 状态滚动更新，为计算下一层做准备
            prev2 = prev1;
            prev1 = current;
        }
        
        return current;
    }
}
