# [0033. 搜索旋转排序数组 (Search in Rotated Sorted Array)]

## 💡 核心直觉：局部有序定理与二分破局
面对 $O(\log n)$ 的时间复杂度硬性要求，破局的核心算法必然是**二分查找**。
但由于数组被“折断”成了两段局部的单调递增区间，传统的全局二分无法直接使用。我们在探讨中明确了最核心的直觉：**无论我们取哪个中间点 `mid`，将数组一分为二后，必定有一半是完全单调递增的（完美的有序区间）！**
解题的大方向就变成了：
1. 找出哪半边是完美有序的。
2. 判断 `target` 是否稳稳落在这个完美有序的区间内。如果在，就在这半边继续二分；如果不在，就去另一半找。

## ⚠️ 致命边界复盘：为什么全用 `<` 或 `>` 会酿成大错？
在第一版草稿中，尝试用三个 `else if` 和严格的大于/小于号来框定逻辑，但这在真实的测试用例中会引发严重的生产事故。我们揪出了两个致命的边界陷阱：

1. **死循环陷阱（当 `left` 和 `mid` 重合时）**：
   对于数组 `[3, 1]` 找 `1`，第一轮计算 `mid = 0 + (1 - 0) / 2 = 0`，此时 `left` 和 `mid` 都指向索引 0（值为 3）。由于使用了严格比较 `nums[left] < nums[mid]`（即 `3 < 3` 必定为 `false`），所有的判断分支全部失效，程序什么都不做直接进入死循环 (TLE)。
   **纠正：** 判断有序时必须加上等号，即 `nums[left] <= nums[mid]`，因为当区间只剩 1-2 个元素时，单一元素自身也是一个合法的“有序区间”。

2. **错失目标陷阱（当 `target` 刚好踩在边界上时）**：
   对于数组 `[4, 5, 6, 7, 0, 1, 2]` 找 `4`。如果判断条件是 `nums[left] < target && target < nums[mid]`，当 `target` 和 `nums[left]` 相等时（`4 < 4` 为 `false`），代码会错误地进入 `else` 分支去右半段寻找，完美错过了正确答案。
   **纠正：** 框定目标是否在“完美区间”内时，必须照顾到“地板”和“天花板”，使用 `>=` 和 `<=` 来锁死区间。

## ⚙️ 工业界真实工程实例：无锁环形队列 (Ring Buffer) 与时间序列检索
跳出纯算法的视野，这个“旋转排序数组”在基础架构和中间件（如 LMAX Disruptor、高并发 APM 监控系统）中有着极其经典的落地场景：
* **GC 梦魇的解法**：为了避免不断创建对象引发 Java 的 Full GC，高并发系统往往预先分配一个固定大小的巨型数组来存储打点数据。
* **物理映射**：当数据写满数组后，写指针会绕回数组头部覆盖老数据。此时，这个按照时间戳追加的数组，就变成了一个**标准的“旋转排序数组”**（写指针当前所在的位置就是旋转的断点）。
* **算法的工业价值**：当监控大盘需要查询某个历史时间戳的快照时，由于数组是局部有序的，使用普通的线性遍历会导致 CPU 飙升甚至超时。正是利用我们手写的 $O(\log n)$ 旋转二分法，才能在百万级的数据中仅用几十次循环瞬间定位到目标，这就是底层算法在工业界拉开性能差距的关键。

## 💻 最终 Java 代码实现 (大厂 Code Review 级别注释)
* **时间复杂度**：$O(\log n)$，标准的二分折半效率。
* **空间复杂度**：$O(1)$，仅使用了三个常数级指针。

```java
class Solution {
    public int search(int[] nums, int target) {
        int left = 0;
        int right = nums.length - 1;
        
        // 【边界说明】必须使用 <=，因为搜索区间是闭区间 [left, right]。
        // 当 left == right 时（即区间内只剩最后一个元素），我们仍需要进循环验证这个元素是否就是 target。
        while (left <= right) {
            // 防溢出写法：等同于 (left + right) / 2。
            // 避免 left 和 right 接近整型最大值时，直接相加导致 int 溢出变为负数。
            int mid = left + (right - left) / 2;
            
            // 命中目标，直接返回
            if (nums[mid] == target) {
                return mid;
            } 
            // 【核心边界 1】判断左半段 [left, mid] 是否是纯粹的单调递增区间
            // 为什么必须用 <= ？
            // 因为当区间只剩 1 个或 2 个元素时，由于 mid 是向下取整，mid 会和 left 重合。
            // 此时 nums[left] == nums[mid]，由于只有一个元素，它自身当然算作"有序"，所以必须包含等于的情况。
            else if (nums[left] <= nums[mid]){
                // 【核心边界 2】目标值 target 是否稳稳落在这个完美的左半段有序区间内？
                // 必须同时满足：>= 地板（nums[left]） 且 <= 天花板（nums[mid]）。
                if (nums[mid] >= target && nums[left] <= target) {
                    right = mid - 1; // 确认在左半段，果断砍掉右边
                } 
                else {
                    left = mid + 1;  // 不在这个完美的左半段，那一定在另一半（可能包含旋转点的右半段）
                }
            }
            // 走到这里，说明 nums[left] > nums[mid]，即左半段发生了旋转。
            // 那么根据局部有序定理，右半段 [mid, right] 绝对是一段完美的单调递增区间。
            else{
                // 【核心边界 3】目标值 target 是否稳稳落在这个完美的右半段有序区间内？
                // 同样需要满足：> 地板（nums[mid]） 且 <= 天花板（nums[right]）。
                // （注：因为第一步已经排除了 nums[mid] == target 的情况，所以这里 nums[mid] 严格 < target 逻辑极其严密）
                if (nums[mid] < target && nums[right] >= target) {
                    left = mid + 1;  // 确认在右半段，果断砍掉左边
                } 
                else {
                    right = mid - 1; // 不在完美的右半段，那必定在包含旋转点的左半段
                }
            }                          
        }
        // 整个循环结束都没有 return，说明数组中不存在 target
        return -1;
    }
}
