## 二、 灵魂拷问：为什么慢指针相遇时绝对走不满一圈？
*(注：这是本次刷题中最具价值的思考，也是大厂面试中极容易被追问的细节)*
很多死记硬背的候选人无法回答这个问题，我们可以用**相对速度**完美证明：
1. **定格入环瞬间**：当慢指针 `slow` 刚到达入环点时，快指针 `fast` 已经在环内了。
2. **极限追击距离**：假设环的周长为 $L$。此时 `fast` 从后方追击 `slow`，它们之间的距离最大也就是 $L - 1$。
3. **降维打击**：以 `slow` 为静止参照物，`fast` 正在以 **1 步/次** 的相对速度向 `slow` 逼近。
4. **得出结论**：`fast` 追上 `slow` 最多只需要 $L - 1$ 次移动。在这段时间内，`slow` 也只走了 $L - 1$ 步，**绝对小于环的周长 $L$**。因此，`slow` 连第一圈都跑不完就会被抓住！

## 三、 核心数学推导与解法沉淀

### 最优解：龟兔赛跑进阶版 (空间复杂度 $O(1)$)

* **变量定义**：
  * $a$ = 起点到入环点的距离
  * $b$ = 入环点到相遇点的距离
  * $c$ = 相遇点回到入环点的距离 (环长 $L = b + c$)
* **代数推导**：
  * 相遇时 `slow` 走的距离：$a + b$
  * `fast` 走的距离：$a + n(b + c) + b$
  * 因为 `fast` 速度是 `slow` 的两倍：$2(a + b) = a + n(b + c) + b$
  * 化简得出核心等式：**$a = (n - 1)(b + c) + c$**
* **魔法操作**：这意味着，**起点到入环点的距离 ($a$)** 刚好等于 **相遇点到入环点的距离 ($c$)** 加上若干圈环长。只要让一个指针从头 `head` 开始走，另一个从相遇点 `slow` 开始走，每次各走 1 步，它们**必定在入环点相遇**！

**核心 Java 代码定稿 (0ms 击败 100%)**：
```java
public class Solution {
    public ListNode detectCycle(ListNode head) {
        ListNode slow = head, fast = head, entry = head;
        while (fast != null && fast.next != null) {
            slow = slow.next;
            fast = fast.next.next;
            if (slow == fast) { // 在环内相遇
                while (slow != entry) { // 寻找入环点
                    slow = slow.next;
                    entry = entry.next;
                }
                return entry;
            }
        }
        return null;
    }
}

```

### 备选解法：哈希标记法 (空间复杂度 $O(N)$)

通过 `HashSet` 记录访问过的节点内存引用。第一个发现 `visited.contains(curr)` 为 `true` 的节点，即为入环点。业务防雷常用，逻辑最直白。

## 四、 工业界真实场景映射 (工程实例)

1. **密码学碰撞破解 (Pollard's rho)**：利用入环点的推导逻辑，在 $O(1)$ 内存下，在伪随机 $\rho$ 路径中寻找导致 Hash 值一样的两个不同明文输入（碰撞点）。
2. **数据库死锁回滚**：不仅检测等待图（Wait-For Graph）是否成环，更通过寻找“入环点”来定位真正引发死锁的环内事务，精确挑选代价最小的 Victim 事务进行 Kill，避免误杀无辜的外围排队事务。
